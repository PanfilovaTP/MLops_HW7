# 1. Использовать Canary Deployment для ML-сервиса

Дата: 2026-04-28

## Статус

Принято

## Контекст

В проекте разворачивается ML-сервис с двумя версиями модели: стабильная `v1.0.0` и новая `v1.1.0`. В исходном коде почти нет полноценной обработки ошибок: если новая версия модели начнет возвращать некорректные ответы или нестабильно обрабатывать входные данные, проблема может сразу затронуть пользователей. Поэтому стратегия деплоя должна снижать риск массового отказа и позволять быстро откатиться на предыдущую версию.

## Рассмотренные варианты

### Blue-Green Deployment

Плюсы:
- две версии сервиса существуют параллельно;
- можно проверить `/health` у новой версии до переключения трафика;
- откат простой: вернуть балансировщик на старую версию.

Минусы:
- при переключении на green новая версия может сразу получить 100% трафика;
- если в новой версии есть скрытая ошибка, влияние на пользователей будет резким;
- не позволяет постепенно измерять качество модели на небольшой доле запросов.

### Canary Deployment

Плюсы:
- новая версия получает сначала небольшую долю трафика, например 10%;
- можно наблюдать ошибки `/predict`, `/health`, задержки и качество модели до полного выката;
- при проблемах можно вернуть 100% трафика на `v1.0.0`;
- хорошо подходит для ML-моделей, потому что качество новой версии может отличаться на реальных данных.

Минусы:
- настройка немного сложнее, чем Blue-Green;
- нужны правила мониторинга и критерии отката;
- при малой доле трафика статистическая оценка качества может занимать больше времени.

### Rolling Deployment

Плюсы:
- постепенная замена инстансов;
- экономит ресурсы.

Минусы:
- хуже подходит для текущей задачи, потому что сложнее быстро сравнивать две версии модели;
- при ошибках часть пользователей уже может попасть на новую версию.

### Shadow Deployment

Плюсы:
- новая версия получает копию реального трафика без влияния на пользователей;
- безопасно для проверки качества.

Минусы:
- требует дополнительной инфраструктуры для зеркалирования запросов;
- не проверяет полный пользовательский сценарий с ответом новой модели.

## Решение

Выбрана стратегия **Canary Deployment**. На первом этапе балансировщик Nginx распределяет трафик между версиями `v1.0.0` и `v1.1.0` в пропорции 90/10. Если `/health`, `/predict` и базовые метрики качества стабильны, доля новой версии увеличивается до 50%, затем до 100%. Если новая версия работает нестабильно, выполняется rollback: весь трафик возвращается на `v1.0.0`.

## Последствия

Положительные последствия:
- риск массового отказа снижен;
- есть быстрый rollback;
- можно сравнивать поведение двух версий ML-сервиса на реальном трафике;
- стратегия хорошо сочетается с A/B-тестированием и мониторингом качества модели.

Отрицательные последствия:
- требуется поддерживать несколько конфигураций Nginx;
- нужен контроль метрик и логов;
- деплой становится сложнее, чем запуск одной версии сервиса.
